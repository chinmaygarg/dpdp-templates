Overview & Risk Classification

This plan establishes mandatory security controls for all DPDP-regulated data. Non-compliance creates direct exposure to ₹50–₹100 Cr penalties under DPDP Section 8 (Information Security violations).

⚠️ CRITICAL:
Personal data = PII (Aadhaar, PAN, phone, email) + KYC docs + transaction history + behavioral data. ALL must be encrypted at rest and in transit.

Data Classification by Risk

Data Class Examples At-Rest Encryption In-Transit Protection Access Control
Tier 1: Identifiers Aadhaar, PAN, passport, driving license AES-256-GCM (mandatory) TLS 1.3 + mTLS for APIs 2FA + role-based (admin only)
Tier 2: KYC/Financial Bank account, salary, credit score, loan details AES-256-GCM (mandatory) TLS 1.3 + IPSec for data lakes 2FA + approval for access
Tier 3: Behavioral Transaction history, browsing, app usage AES-256-GCM (mandatory) TLS 1.2+ Role-based access

India-Specific Regulatory Requirements

These rules are non-negotiable for India-regulated organizations. Violations trigger DPBI escalation and RBI penalties.

Rule 1: Aadhaar VID Compliance (MANDATORY)

✗ NEVER STORE 12-DIGIT AADHAAR NUMBER
✓ ALWAYS USE AADHAAR VIRTUAL ID (VID)

Rule: 12-digit Aadhaar number must NEVER be persisted in any database, cache, log file, or backup. Only Virtual ID (16-digit, one-time) may be stored. If Aadhaar matching is needed for deduplication, hash the Aadhaar number (do not store the original).
Scenario ❌ WRONG ✅ CORRECT
User authenticates via Aadhaar Store 12-digit Aadhaar in DB Request VID from UIDAI, store VID only
Verify duplicate Aadhaar Query DB for matching 12-digit number Hash incoming Aadhaar, compare with stored hash (no plaintext storage)
Logs / Debugging Log full Aadhaar in application logs Redact Aadhaar in logs (e.g., AADHAAR_REDACTED)

Rule 2: PAN Masking in All Outputs (MANDATORY)

PAN MASKING RULE: PAN must be masked to last 4 digits (e.g., XXXXXXX1234) in:
✓ All logs and debug output
✓ API responses and frontend displays
✓ Exports (CSV, PDF, Excel)
✓ Audit trail and file backups
✓ Error messages and exceptions

Full PAN is stored in DB (encrypted at rest), but NEVER exposed in any output except payment processing APIs.
Context Masking Rule Example
Admin dashboard display Last 4 only XXXXXXX1234
API response (public) Masked XXXXXXX1234
Tax filing API (restricted) Full PAN (internal use only) AAAPP1234A
Logs / Console output Redacted [PAN-REDACTED]

Encryption Standards & Key Management

1. At-Rest Encryption (Database & Storage)

Component Algorithm Key Length Implementation
Database (PII tables) AES-256-GCM 256 bits Column-level (per sensitive field) or full-disk encryption (PostgreSQL pgcrypto)
File Storage (S3/Blob) AES-256 (server-side) 256 bits S3 SSE-KMS (AWS KMS) or Azure Key Vault
Backups AES-256 256 bits Database-native backup encryption (PostgreSQL pg_dump --with-encryption)
Cache (Redis) AES-256 256 bits Redis encryption at rest (Redis 7+) + TLS in transit

2. In-Transit Encryption (Network)

Channel Protocol Minimum Version Cipher Strength
Public APIs (web, mobile) HTTPS (TLS) TLS 1.3 (TLS 1.2 minimum) AEAD ciphers (AES-256-GCM, ChaCha20)
Internal APIs (service-to-service) mTLS (mutual TLS) TLS 1.3 (TLS 1.2 minimum) Client & server certificates required
Database connections TLS / SSL TLS 1.2+ PostgreSQL sslmode=require
Admin consoles (VPN) VPN + HTTPS VPN IKEv2 with AES-256 End-to-end encryption

3. Key Management

Key Rotation Schedule:
✓ Database encryption keys: Rotate quarterly (every 90 days)
✓ TLS certificates: Rotate annually + before expiry (with 30-day lead time)
✓ API keys / OAuth tokens: Rotate semiannually or on personnel change
✓ Master key (key-encryption-key): Stored in AWS KMS / Azure Key Vault, rotate annually

Key Storage: Never hardcode keys in code. Use managed key services (AWS KMS, Azure Key Vault, HashiCorp Vault) with audit logging.

Access Control & MFA Enforcement

1. Multi-Factor Authentication (MFA) Mandate

Role MFA Required? Accepted Methods Enforcement
Admin (DPO, Legal, Compliance) ✅ YES (MANDATORY) TOTP (Google Authenticator) + SMS backup OTP Enforce at login + every 4 hours
Approver (DPA Review, DPIA Sign-off) ✅ YES (MANDATORY) TOTP + SMS OTP Enforce at login + before sensitive action
Regular users (viewer, analyst) ⚠️ RECOMMENDED TOTP or push notification Default enabled, can be disabled per-user policy
Data Principals (external) ✅ YES (for DSAR) OTP to email / SMS, account login + DOB verification, Aadhaar VID Mandatory before DSAR completion

2. Role-Based Access Control (RBAC)

Least Privilege Principle: Each role has minimum permissions to perform their function.

Permission Assignment:
• Admin: Full system access (users, config, breach response)
• DPO: Read + audit logs, DPIA approval, compliance reports
• Legal: Notice + consent + DPA review
• Analyst: Read-only reports, no sensitive data exposure
• Vendor: API-only access (no web console), limited to their data scope

Quarterly Access Review: Every 90 days, audit role assignments and remove excessive permissions.

Database Activity Monitoring (DAM)

DAM monitors all interactions with sensitive databases. Mandatory alert rules for DPDP Section 8 compliance.

DAM Alert Rules

Alert Rule Trigger Condition Action
Bulk Data Export >1000 rows exported in <5 min Alert CISO, require 2FA confirmation
Schema Changes ALTER TABLE on PII tables outside maintenance window Block query, alert DBA + Security
After-Hours Access Database access between 10 PM–6 AM (outside business hours) Trigger MFA re-verify, log full query
Privilege Escalation User switches role from analyst to admin Block switch, require DPO approval
Anomalous Query Query executes >100x slower than baseline (potential DoS) Kill query, alert DBA
Failed Auth Attempts >5 failed logins in 15 min from single source Lock account, alert Security
Unencrypted Data Access Query returns PII without encryption context Log query details, flag for audit

Zero Trust & Least Privilege Architecture

Zero Trust principle: Never trust, always verify. All access requests (user-to-system, service-to-service) require explicit verification.

Zero Trust Implementation Checklist

  • Identity Verification: Every request includes identity proof (MFA token, mutual TLS certificate, API signature). No assumption of trust based on network location.
  • Least Privilege Access: Users have minimum permissions to perform job function. Permissions reviewed quarterly. JIT (Just-In-Time) elevated access for admin tasks (auto-revoke after 2 hours).
  • Micro-Segmentation: Database, APIs, file storage are segregated by function. Breach in one service cannot spread to another.
  • Encryption Everywhere: All data in transit encrypted. Sensitive data at rest encrypted.
  • Audit Trail: All access logged. Changes to permissions logged. Logs immutable (append-only, signed).
  • Continuous Monitoring: Behavioral analysis detects anomalous access patterns. Alerts trigger incident response.

Quarterly Access Review Process

Every 90 days:
1. Security team exports all active user accounts + their current permissions
2. Department heads review assignments and remove inactive users
3. For each user with sensitive access, re-verify business need
4. Remove permissions that are no longer required
5. DPO signs off on access matrix
6. Board Risk Committee receives summary (# users removed, # permissions revoked)
7. Findings documented in Audit Findings Report (T-28)
