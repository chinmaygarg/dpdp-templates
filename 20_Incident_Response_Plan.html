Incident Response Plan
Data Breach Detection, Response & Dual-Timer SLA (DPDP Section 8 + CERT-In 2022)
Version: 2.0 (Updated April 2026) | Effective Date: 13 May 2027 | Status: Active

Overview

Under DPDP Act 2023 Section 8, organizations must implement technical and organizational measures for data security. Breaches require notification to the Data Protection Board of India (DPBI) within 72 hours. Additionally, ALL cybersecurity incidents (including those without personal data) must be reported to CERT-In within 6 hours per CERT-In Direction 2022, as CERT-In is the primary cybersecurity regulator.

⚠️ CRITICAL DISTINCTION:
• CERT-In (6-hour SLA): All cybersecurity incidents (data breach, ransomware, intrusion, DDoS, unauthorized access)
• DPBI (72-hour SLA): Only incidents involving personal data that create compliance breach risk
Many organizations only address DPBI and miss CERT-In entirely, creating regulatory exposure.

Dual-Timer Breach Command Structure

Data breaches trigger two parallel 72+ hour response timelines managed by different regulators. This timeline tracks both from incident detection through post-incident review.

Timeline Action Owner Authority Status
T+0 (Hour 0) Breach detected / reported / alert triggered Security Team / SOC — Incident declared
T+1 (Hour 1) Incident Commander activated; War Room established; Initial triage IR Lead / CISO — Incident team assembled
T+2 (Hour 2) Breach classification completed: Type (data leak, ransomware, intrusion, etc.) + Severity (Critical/High/Medium/Low) IC / Security Team — Classification locked
T+6 (Hour 6) CERT-In Notification Filed (Form CC-5) — includes incident type, vector, systems affected, data involved (yes/no) CISO / IR Lead CERT-In
(6-hour SLA)		 MANDATORY
T+24 (Hour 24) Containment confirmed; affected data principals identified (count + data categories) IC / DPO / Forensics — Scope confirmed
T+48 (Hour 48) DPO signs off on DPBI notification draft (contains: data involved, affected principals, root cause, remediation plan) DPO — Draft ready
T+72 (Hour 72 / 3 days) DPBI Notification Filed (Form: Breach Notification Form) — includes full incident details, data involved, remediation DPO / Compliance DPBI
(72-hour SLA)		 MANDATORY
T+30d (30 days) Post-Incident Review (PIR) completed; root cause analysis, lessons learned, control improvements documented IC / CISO / DPO — Incident closed
✅ CRITICAL SUCCESS:
Both CERT-In (T+6) and DPBI (T+72) notifications must be filed on time. Miss either and the organization faces ₹50 Cr penalties + regulatory action.

Breach Classification Matrix

All cybersecurity incidents must be classified by Type and Severity. Severity determines escalation, resources, and communication approach.

Type Description Severity if Data Involved CERT-In Required? DPBI Required?
Data Exfiltration (Leak) Unauthorized access + copy of PII/data outside organization CRITICAL ✅ Yes (T+6) ✅ Yes (T+72)
Ransomware Malware encryption of files; extortion demand; operational disruption CRITICAL ✅ Yes (T+6) ✅ Yes (if data accessed)
Unauthorized Access Login compromise, privilege escalation, insider threat access to data HIGH ✅ Yes (T+6) ✅ Yes (if data accessed)
DDoS Attack Service disruption; no data compromise (unless combined with breach) MEDIUM ✅ Yes (T+6) ❌ No (unless data involved)
Malware Infection Virus/worm/trojan on endpoint; intent/impact depends on payload HIGH ✅ Yes (T+6) ✅ Yes (if keylogger/credential stealer)
Configuration Error / Data Exposure Cloud bucket/DB left public; backup exposed; unintended data visibility HIGH ✅ Yes (T+6) ✅ Yes (T+72)
Failed Authentication / Brute Force Bulk failed login attempts; credential stuffing; no successful access LOW ✅ Yes (T+6) ❌ No (unless successful breach)
Insider Threat / Data Exfiltration Employee/contractor intentionally copies/transfers data outside organization CRITICAL ✅ Yes (T+6) ✅ Yes (T+72)

CERT-In Notification (6-Hour SLA)

CERT-In 2022 Direction requires reporting of ALL cybersecurity incidents to the Indian Computer Emergency Response Team (CERT-In) within 6 hours. This includes incidents that may not involve personal data but represent cybersecurity threats.

What Must Be Reported to CERT-In

  • ✅ Data breaches (exfiltration of PII/data)
  • ✅ Ransomware attacks
  • ✅ Unauthorized access (including failed brute-force attempts)
  • ✅ Malware infections
  • ✅ DDoS attacks
  • ✅ Website defacement
  • ✅ Phishing/social engineering attacks (if successful compromise resulted)
  • ✅ Insider threat incidents
  • ✅ Configuration errors exposing data/systems

Incident Commander Role & Authority

A single Incident Commander (IC) leads the breach response. The IC has full authority to: make decisions, reallocate resources, and communicate with external stakeholders (CERT-In, DPBI, media).

Incident Commander Responsibilities

Phase IC Responsibility
Detection (T+0-1) Convene War Room; activate IR team; define incident scope; ensure SIEM/forensics logging active
Classification (T+1-2) Classify incident type + severity; determine regulatory obligations (CERT-In, DPBI)
CERT-In Notification (T+2-6) Prepare CERT-In Form CC-5; coordinate with CISO; file by T+6 hour deadline
Containment (T+2-24) Execute containment playbook; isolate affected systems; preserve evidence; prevent data exfiltration
Scope & Impact (T+6-24) Quantify affected Data Principals; identify data categories; assess harm likelihood
Data Principal Notification (T+24-72) Approve notification language; coordinate with Legal; ensure timely, accurate communication
DPBI Notification (T+48-72) Ensure DPO prepares & files DPBI notification by T+72 deadline
Recovery & Remediation (T+30-90 days) Restore systems; apply patches; implement corrective controls; conduct PIR

Incident Commander Escalation Authority

  • Authority to isolate systems — can disconnect networks, shut down services without approval
  • Authority to reallocate resources — can reassign staff from other projects
  • Authority to communicate externally — with CISO, Legal, PR, regulators
  • Authority to defer non-critical work — can pause non-critical operations
  • Reports to: CISO (operational) + CEO/Board (strategic escalation)

Incident Commander Selection Criteria

  • ✅ Technical expertise: Deep knowledge of systems affected (DB, network, application)
  • ✅ Decision-making ability: Can make rapid decisions under pressure
  • ✅ Leadership: Can command respect from cross-functional teams
  • ✅ Experience: Has managed previous incidents or crisis situations
  • ✅ Regulatory knowledge: Familiar with DPDP, CERT-In, DPBI requirements
  • ✅ Pre-designated: Named in advance (not chosen during incident — avoids delays)

Containment Playbook (6 Scenarios)

Each breach type requires a specific containment approach. Below are 6 common scenarios with step-by-step containment procedures.

Scenario 1: Data Exfiltration (Unauthorized Download)

Severity: CRITICAL | CERT-In: Yes (T+6) | DPBI: Yes (T+72)
  1. Immediate (T+0-30min): Isolate database server from network; halt all outbound connections; enable packet capture
  2. T+30-60min: Identify compromised account; revoke credentials; check AWS CloudTrail / DB audit logs for volume downloaded
  3. T+1-2hr: Identify all data exported; generate list of affected Data Principals; assess if encryption was bypassed
  4. T+2-6hr: Preserve forensics: copy database, log files, filesystem to external drive; calculate hash for chain of custody
  5. T+6hr: File CERT-In notification with data volume, principals affected, vector
  6. T+24hr: Verify containment (no further exfiltration); confirm all affected principals counted
  7. T+48hr: Begin Data Principal notification (email, portal, optional credit monitoring); DPO approves DPBI draft
  8. T+72hr: File DPBI notification

Scenario 2: Ransomware Encryption

Severity: CRITICAL | CERT-In: Yes (T+6) | DPBI: Yes (if data compromised)
  1. Immediate (T+0-15min): Isolate affected server(s) from network; disconnect backup systems to prevent encryption spread
  2. T+15-30min: Disable RDP/SSH access; kill suspicious processes; preserve memory dump for analysis
  3. T+30-60min: Identify infection vector (email attachment, vulnerability, brute force); lock affected user accounts
  4. T+1-2hr: Assess which databases/files encrypted; determine if encrypted data includes personal data
  5. T+2-6hr: Preserve forensics: copy disk image, logs, memory dump; do NOT pay ransom without legal counsel
  6. T+6hr: File CERT-In notification; assess if personal data was accessed before encryption
  7. T+24-48hr: Begin recovery from clean backups (if available); patch vulnerability that allowed entry
  8. T+48hr: If personal data accessed: Data Principal notification + DPBI draft
  9. T+72hr: File DPBI notification (if applicable)

Scenario 3: Compromised Vendor / Third-Party Breach

Severity: HIGH | CERT-In: Yes (T+6) | DPBI: Yes (if data of your principals accessed)
  1. Immediate (T+0-1hr): Isolate vendor system access; revoke API keys, service accounts; notify vendor of incident
  2. T+1-2hr: Identify what data vendor accessed (check API logs, DB audit logs); determine if your Data Principals' data exposed
  3. T+2-6hr: Review Data Processing Agreement (T-24); vendor must be obligated to notify you within 24hrs
  4. T+6hr: File CERT-In notification; include vendor details and data involved
  5. T+24hr: Receive incident details from vendor; assess impact to your Data Principals
  6. T+48hr: If your principals affected: prepare Data Principal notification; DPO approves DPBI draft
  7. T+72hr: File DPBI notification if your principals' data accessed
  8. Post-incident: Re-assess vendor risk (T-15 Vendor Risk Assessment); consider contract termination if SLA breached

Scenario 4: Database / Cloud Configuration Error Exposing Data

Severity: HIGH | CERT-In: Yes (T+6) | DPBI: Yes (T+72)
  1. Immediate (T+0-15min): Make database/bucket private; revoke public access; document current state (screenshot, metadata)
  2. T+15-60min: Identify when misconfiguration occurred; determine if any unauthorized access during exposure window
  3. T+1-2hr: Check access logs (AWS CloudTrail, Azure audit logs) for non-organization IPs during exposure period
  4. T+2-6hr: Quantify records exposed; identify affected Data Principals; assess likelihood unauthorized parties accessed data
  5. T+6hr: File CERT-In notification; describe misconfiguration, exposure window, access indicators
  6. T+24hr: Complete forensics; determine if data was actually accessed by unauthorized parties or only discoverable
  7. T+48hr: If confirmed access: prepare Data Principal notification; DPO reviews risk assessment and DPBI draft
  8. T+72hr: File DPBI notification (if data accessed or risk is high)

Scenario 5: Insider Threat / Employee Data Theft

Severity: CRITICAL | CERT-In: Yes (T+6) | DPBI: Yes (T+72)
  1. Immediate (T+0-30min): Revoke employee credentials immediately; disable VPN, email, system access
  2. T+30-60min: Identify what data employee accessed/copied; check file transfer logs, email logs, cloud uploads
  3. T+1-2hr: Secure employee laptop/phone; preserve device for forensics; check personal cloud accounts (Google Drive, Dropbox) for uploads
  4. T+2-6hr: Notify Legal/HR; determine if police report needed (criminal data theft); preserve chain of custody for evidence
  5. T+6hr: File CERT-In notification; describe insider threat, data accessed, exfiltration method
  6. T+24-48hr: Data Principal notification if personal data confirmed exfiltrated; assess if data can be recovered (demand return, legal action)
  7. T+48hr: DPO reviews DPBI draft; if data exfiltrated: file DPBI notification at T+72
  8. Post-incident: Implement access controls improvements (principle of least privilege, activity monitoring)

Scenario 6: Failed Authentication / Brute Force (No Successful Access)

Severity: LOW-MEDIUM | CERT-In: Yes (T+6) | DPBI: No (unless breached)
  1. Immediate (T+0-30min): Monitor for successful login; block attacking IPs; enable 2FA/MFA for targeted accounts
  2. T+30-60min: Verify no successful logins during attack period; check logs for lateral movement
  3. T+1-2hr: Identify attack source (IP geolocation, credential list source); determine if targeted attack or random
  4. T+2-6hr: Document in incident log; no data accessed, so breach containment not required
  5. T+6hr: File CERT-In notification (security incident even without data compromise); describe attack pattern, detection, mitigation
  6. T+24hr+: No DPBI notification required (no data breach); implement controls: IP-based rate limiting, adaptive auth, login alerting

Evidence Preservation & Forensics

Data breaches often lead to legal action (Data Principals suing, DPBI enforcement, police investigation). All evidence must be preserved with unbroken chain of custody to be admissible in court.

What to Preserve

Evidence Type How to Preserve Retention Period
Database/File Contents Create forensic image (disk-level copy); compress with tar/gzip; calculate MD5 + SHA-256 hash Until litigation resolved or DPBI order fulfilled
System Logs Export from SIEM; export from database audit logs, web server logs, OS logs; sign with timestamp + hash Until litigation resolved
Network Packets Enable packet capture on IDS/network monitoring; save PCAP files; timestamp each Until forensics complete
Memory Dump Use `dd`, `memdump`, or hypervisor snapshot; preserve before system restart; calculate hash Until forensics complete
Compromised Endpoint Create full disk image before wiping; do NOT use device after compromise (limits evidence contamination) Until forensics complete
Email / Communication Records Export from email server; disable automatic purge/archival; include headers for authentication Until litigation resolved
Chain of Custody Log Document: Who collected, what collected, when, where stored, who accessed, purpose of access Permanent (admissibility depends on this)

Chain of Custody Requirements

For evidence to be admissible in court, every transfer of custody must be documented:

  • Who collected it: Name, title, timestamp
  • What was collected: Specific files, records, systems
  • Where stored: Encrypted drive, secure facility, locked cabinet
  • Who accessed it: Every person who touched the evidence must be logged
  • Purpose of access: Why was it examined/reviewed
  • Time in custody: From T+0 until handed to court/external forensics

SIEM Alert Rules & Triggers

Security Information and Event Management (SIEM) systems must be configured with rules that detect data breach indicators. Below are critical rules that should trigger immediate incident response.

Alert Name Trigger Condition Severity Action
Large Data Export >100 records selected + exported in <5 minutes; OR >1 GB exported in single session CRITICAL Page on-call; freeze account; begin forensics
Unauthorized Database Access Login from non-whitelisted IP; multiple failed attempts then 1 success; access during non-business hours CRITICAL Page on-call; revoke session; query for data accessed
Credential Stuffing >10 failed login attempts from same IP in <10 minutes targeting multiple user accounts HIGH Block IP; enable CAPTCHA; notify affected users
Privilege Escalation Standard user executes sudo/admin command; or user suddenly gains DBA role without approval CRITICAL Page on-call; suspend user; audit all commands by user
Bulk Data Deletion DELETE >1000 records in single query; DROP table command; or file deletion of >1 GB in <1 minute CRITICAL Page on-call; verify with system owner; restore from backup if not authorized
Ransomware File Pattern Detection of .locked, .encrypted, .ransom file extensions; or ransom note files created; or high file modification rate CRITICAL Isolate system immediately; page on-call; begin incident response
Suspicious Outbound Connection Database server connecting to external IP (not approved); large data transfer outbound from DB server HIGH Page on-call; block connection; investigate destination IP
Malware Detection Antivirus/EDR detects known malware signature; or suspicious binary execution from temp directory CRITICAL Isolate infected system; quarantine; begin forensics; page on-call
Unauthorized API Access API key used from non-whitelisted IP; API key querying unusual endpoints; unusually high volume API calls HIGH Revoke key; page on-call; audit API calls; check for data access
Access Log Deletion System logs cleared/truncated; audit trail deleted; SIEM logs missing gaps; history cleared CRITICAL Page on-call; begin forensics (may indicate cover-up); isolate system; preserve memory dump

Annual Tabletop Exercise

Breach response must be practiced annually via tabletop exercises to ensure all team members understand their roles and timelines are achievable.

Tabletop Exercise Structure

Duration: 2-3 hours | Frequency: Annually (or after major incidents) | Participants: CISO, IR Lead, DPO, Legal, PR, CTO, CFO, incident responders

Scenario Example: Ransomware Attack with Data Exfiltration

  1. Scenario Brief: "It's 9 AM Tuesday. Security SOC discovers ransomware infection on 3 database servers. Ransom note appears. First detection was 2 AM (7 hours ago). Encryption detected on /home, /data, and backup drives."
  2. Facilitator-Led Discussion:
    • Who is the Incident Commander? (should be pre-designated)
    • What's the first action? (isolate infected servers from network)
    • How do we verify forensic evidence chain of custody? (document who collected what, when, where)
    • When do we notify CERT-In? (T+6 hours — we've already missed deadline; now what?)
    • How many Data Principals affected? (must query unencrypted backups or replicas to estimate)
    • What's the DPBI notification timeline? (T+72 from detection = by 2 AM Thursday)
    • Do we pay the ransom? (Legal + Leadership decision; FBI/CERT-In consultation)
    • Data Principal notification plan? (email + portal notification within 72 hours of DPBI filing)
  3. Simulation of Timeline Execution: Walk through T+0, T+1, T+2, T+6, T+24, T+48, T+72 milestones. Discuss blockers and how each is resolved.
  4. Lessons Learned Discussion: What surprised you? What would you do differently? What controls need improvement?
  5. Document Outcomes: Record decisions, timeline feasibility, control gaps, training needs.
  6. Improvement Actions: Create follow-up tasks for any process improvements identified.

Implementation Checklist

✅ PRE-INCIDENT PREPARATION
  • ☐ Incident Commander: Pre-designated; has authority to isolate systems, reallocate resources, communicate with regulators
  • ☐ Incident Response Team: Named (Security/Forensics, DPO, Legal, PR, CTO); contact list shared with all members
  • ☐ War Room Setup: Video call, shared documents, status tracking; conference line reserved
  • ☐ CERT-In Account: Organization registered with CERT-In; knows how to submit CC-5 form
  • ☐ DPBI Portal Access: DPO has access to (anticipated) DPBI incident notification portal
  • ☐ Breach Notification Template: Legal-reviewed Data Principal notification email prepared
  • ☐ SIEM Configured: All 10 breach detection rules above active and monitored 24/7
  • ☐ Forensics Tools Ready: dd, memdump, PCAP tools installed; external secure drives available
  • ☐ Evidence Repository: Secure, encrypted storage for forensic images and logs
  • ☐ Backup Verification: Confirm offline backups exist and are restorable (test restore at least quarterly)
  • ☐ Network Segmentation: PII databases isolated from internet-facing systems; egress filtering in place
  • ☐ Encryption in Transit & Rest: All PII encrypted TLS 1.3 in transit; AES-256-GCM at rest
ℹ️ DURING-INCIDENT EXECUTION
  • ☐ T+0-1hr: Incident Commander activates; War Room convened; incident logged with timestamp
  • ☐ T+0-2hr: Initial triage: what systems affected, what data involved, scope assessment
  • ☐ T+2hr: Classification completed: type (data leak, ransomware, etc.) + severity (Critical/High/Medium/Low)
  • ☐ T+2-6hr: Evidence preservation started: SIEM logs, database audit logs, system logs captured with hashes
  • ☐ T+6hr: CERT-In notification filed (Form CC-5) — includes incident type, vector, systems, data (yes/no)
  • ☐ T+6-24hr: Containment: affected systems isolated, credentials revoked, unauthorized access stopped
  • ☐ T+24hr: Impact assessment complete: affected Data Principals counted, data categories identified
  • ☐ T+48hr: DPO reviews DPBI notification draft; Legal approves Data Principal notification language
  • ☐ T+72hr: DPBI notification filed (if personal data involved) — includes full incident details, remediation plan
  • ☐ T+72-96hr: Data Principal notifications sent (email + portal); optionally credit monitoring offered
POST-INCIDENT RECOVERY
  • ☐ T+7-30 days: Systems restored from clean backups or patched; recovery verified
  • ☐ T+30 days: Post-Incident Review (PIR) completed; root cause analysis, lessons learned documented
  • ☐ T+30 days: Control improvements implemented: patched vulnerability, hardened access controls, updated policies
  • ☐ T+30 days: Board/Leadership briefing: incident summary, impact, remediation, cost
  • ☐ T+60 days: CERT-In follow-up response (if CERT-In requested additional info)
  • ☐ T+90 days: Vendor/Processor notification (if third-party involvement); contract review for SLA breaches
  • ☐ Annual: Tabletop exercise to test breach response readiness
