Data Correction & Erasure SOP
Standard Operating Procedure for Data Principal Requests (DPDP Sections 12, 13)
Version: 2.0 (Updated April 2026) | Effective Date: 13 May 2027 | Status: Active

Overview

Under DPDP Act 2023 Section 12, Data Principals have unconditional rights to: (1) Correct inaccurate/incomplete data, (2) Erase data (with limited exceptions). This SOP covers the operational procedures, timelines, and most importantly: propagation to processors and third parties.

⚠️ CRITICAL COMPLIANCE GAP:
Many organizations process the correction/erasure request internally but fail to propagate the change to all processors and third parties. DPDP Section 12(4) requires the organization to ensure all processors and recipients of the data are notified and take action. This is a direct penalty exposure.

Correction vs Erasure: Two Distinct Rights

Section 12 grants two separate rights. Data Principals can request either one independently.

Right DPDP Section Definition SLA Exceptions
Correction 12(1)(a) Fix inaccurate, incomplete, or outdated personal data 30 days None — must always correct if request is valid
Erasure 12(1)(b) Delete data permanently (if no longer needed for stated purpose) 30 days Legal obligation, pending litigation, RBI records, DPIA findings

SLA Timeline (30 Days)

Time Action Owner
T+0 (Day 0) Request received (correction or erasure form submitted) Grievance Team
T+1 (Day 1) Assign request ID; log in correction/erasure tracker Grievance Team
T+3 (Day 3) Verify Data Principal identity; assess feasibility of request Privacy Team
T+15 (Day 15) Execute correction/erasure in own systems; prepare processor notification Data Team / DPO
T+20 (Day 20) Notify all processors + third parties of correction/erasure DPO
T+25 (Day 25) Receive confirmation from processors that change has been applied DPO
T+30 (Day 30) 🔴 DEADLINE: Send completion notice to Data Principal Compliance Officer

Correction Process (7 Steps)

Step 1: Receive Correction Request

Data Principal submits Correction Form (T-13) specifying:

  • What data is inaccurate/incomplete? (e.g., "My address is listed as Mumbai but I moved to Bangalore")
  • What is the correct information? (e.g., "New address: [full address]")
  • Supporting evidence (optional but recommended)

Step 2: Verify Identity & Log Request

Grievance Team:

  • Verify Data Principal identity (same as DSAR identity verification: account login, OTP, VID, etc.)
  • Assign Request ID (e.g., COR-2026-04-15-001)
  • Log in Correction Tracker table with: request ID, principal ID, data field, old value, new value, request date
  • Send acknowledgement email with Request ID and expected completion date (within 30 days)

Step 3: Assess Feasibility & Authority

Privacy Team:

  • Is the correction request legitimate? E.g., if principal claims address is wrong, request proof of new address (e.g., utility bill)
  • Can we correct it? E.g., if data comes from an authoritative source (government, bank), we may need principal to update at source first
  • Decision: Approve or request additional information; document reasoning

Step 4: Search All Records

Data Team:

  • Query ALL systems where the data appears (not just current database, but archives, backups, logs, etc.)
  • Example: If correcting phone number, search: SELECT * FROM data_principals WHERE principal_id = ? AND phone_number = [old_number]
  • Document all locations where old data found

Step 5: Execute Correction Internally

Data Team:

  • Update all records with new information
  • Update database: UPDATE data_principals SET phone_number = [new_number] WHERE principal_id = ?
  • Update backups (if backup restore is within retention period)
  • Log the correction in audit trail: who made change, when, previous value, new value, requester ID
  • Do NOT delete old value — keep audit trail for compliance

Step 6: Notify Processors & Third Parties (PROPAGATION)

DPO (CRITICAL):
This is the most-overlooked requirement. Section 12(4) requires you to inform ALL processors and third parties of the correction.

  • Identify all processors who have a copy of the data (check Data Processing Agreement T-24 for processor list)
  • Send standardized notification email to each processor:

Subject: Data Correction Notification — Request ID [COR-ID] — Principal [PRINCIPAL_ID]

Dear [Processor Name],

We are notifying you of a data correction request we received and approved under DPDP Act 2023, Section 12.

Correction Details:
• Data Principal ID: [PRINCIPAL_ID]
• Data Field: [FIELD_NAME] (e.g., "Phone Number")
• Old Value: [OLD_VALUE]
• New Value: [NEW_VALUE]
• Request ID: [COR-ID]

Action Required:
Please update your records with the new value above within 5 business days. Please confirm completion by replying to this email with: confirmation date and system(s) updated.

No Further Action on Our Part:
You do NOT need to notify the Data Principal separately; we will handle that communication.

Regards,
[Organization Name] — Data Protection Team

Step 7: Confirm & Close Request

Compliance Officer:

  • Once all processors have confirmed correction applied, send completion notice to Data Principal
  • Include: confirmation that correction applied, processors notified, request ID, date completed
  • Mark request as "Completed" in tracker

Erasure Process (8 Steps)

Erasure is more complex than correction because it requires: (1) evaluation of legal exceptions, (2) identification of all copies, (3) permanent deletion, and (4) propagation to all processors.

Step 1: Receive Erasure Request

Data Principal submits Erasure Request Form (T-12 or T-13 can be extended for both). Request can include specific data categories or "all my data".

Step 2: Verify Identity & Log

Same as Correction Step 2. Assign Erasure Request ID (e.g., ERA-2026-04-15-001).

Step 3: Identify Legal Exceptions

DPO / Legal (CRITICAL): Erasure can be denied if:

  • ☐ Legal obligation to retain (e.g., RBI: 8 years for loan records, 10 years for KYC; Income tax: 8 years)
  • ☐ Litigation hold / pending legal dispute (DPO must document hold)
  • ☐ DPIA finding that processing is necessary (document decision + reasoning)
  • ☐ Principal consented to extended retention (document consent agreement)
  • ☐ Data needed to establish, exercise, or defend legal claim
Decision: If exception applies, send justified rejection letter to principal. If no exception, proceed to erasure.

Step 4: Search All Records (All Copies)

Data Team:

  • Search EVERY system: production database, archives, backups, cache servers, logs, external services
  • Identify which systems have which copies: active DB, MySQL backups, S3 archives, CloudFlare cache, third-party APIs, etc.
  • Create deletion checklist with system names and data locations

Step 5: Prepare Deletion Method

Erasure must be PERMANENT — not just marked as deleted. Methods:

  • ☐ Database PERMANENT DELETE: Execute DELETE FROM data_principals WHERE principal_id = ? (not soft-delete)
  • ☐ Backup deletion: Remove old backup tapes/snapshots that contain the data (or isolate them)
  • ☐ Cache clearing: Flush cache servers (Redis, Memcached, CDN) with the principal's data
  • ☐ Log purging: If logs contain PII, redact or delete from log storage
  • ☐ Third-party API: If data synced to external API (Salesforce, Google Analytics), call deletion endpoint or revoke data access
  • ☐ Archive disposal: If archival media (tapes, hard drives) contain data, either physically destroy or cryptographically wipe

Step 6: Execute Erasure (Irreversible)

Data Team with Audit Trail:

  • Execute DELETE in production database for principal's data
  • Log in audit trail (with cryptographic hash) that data was deleted: principal ID, data categories, deletion timestamp, authorized by [person]
  • Do NOT log the actual data values (they're deleted), just that deletion occurred
  • For backups: decide whether to restore and re-delete (if within retention window) or just let them expire naturally (if acceptable by DPO)

Step 7: Notify Processors & Third Parties (PROPAGATION)

DPO (CRITICAL): Same as Correction Step 6, but for erasure:

Subject: Data Erasure Notification — Request ID [ERA-ID] — Principal [PRINCIPAL_ID]

Dear [Processor Name],

We are notifying you of a data erasure request we received and approved under DPDP Act 2023, Section 12.

Erasure Details:
• Data Principal ID: [PRINCIPAL_ID]
• Data Categories to Erase: [List: "Contact information", "Transaction history", "All data"]
• Request ID: [ERA-ID]

Action Required:
Please PERMANENTLY DELETE all personal data for this principal from your systems within 5 business days. This includes: active records, backups, archives, and caches.

Please confirm completion by replying to this email with:
• Confirmation that data has been deleted
• Systems from which data was deleted
• Completion date

If you cannot delete some data due to legal retention requirements, please document the exception and explain the legal basis.

Regards,
[Organization Name] — Data Protection Team

Step 8: Confirm & Close Request

Compliance Officer:

  • Once all processors have confirmed data deleted, send completion notice to Data Principal
  • Include: confirmation of erasure, data categories deleted, processors notified, request ID, completion date
  • Mark request as "Completed" in tracker

Processor & Third-Party Propagation (CRITICAL COMPLIANCE)

DPDP Section 12(4) states: "Where the personal data has been shared with other data fiduciaries or data processors, the data fiduciary shall inform them of the request for correction or erasure..."

⚠️ THIS IS A COMMON VIOLATION:
Many organizations correct/erase the data in their own systems but forget to notify processors. DPBI fines organizations for failing to propagate.

Processor Identification Checklist

DPO must maintain a live list of all processors:

  • ☐ Data Processing Agreement (T-24) lists all processors? (Update if not)
  • ☐ For each processor, document: name, data access, contact email, processor point of contact
  • ☐ Sub-processors: document all sub-contractors (e.g., if processor uses cloud storage, that's a sub-processor)
  • ☐ Third parties: identify recipients who are NOT processors but have received data (partners, vendors, analytics)

Standardized Processor Notification Process

  1. Create processor notification list (from T-24 Processor Register)
  2. For each processor, send standardized email (templates above for correction and erasure)
  3. Request confirmation of action taken within 5 business days
  4. Track confirmations in Correction/Erasure Tracker
  5. If processor does not confirm after 7 days, escalate to DPO and consider contract termination
  6. Document all notifications and confirmations in audit trail for DPBI compliance proof

Sub-Processor Notification

Important: The Data Processing Agreement (T-24) must specify that the processor will:

  • Notify their sub-processors of your correction/erasure request
  • Ensure sub-processors also comply with the request within the same timeline
  • Provide confirmation that sub-processors have complied
Otherwise, you cannot verify that sub-contractors have deleted the data.

Exceptions to Erasure (Legal Holds, Retention)

DPDP Section 12(3) specifies that erasure can be denied in limited circumstances. Document each exception thoroughly.

Exception Documentation Requirements

  • Legal Hold: Document court case number, litigation status, why data is needed for legal claim
  • RBI Retention: Document RBI rule, record type, retention period, date when data can be deleted
  • Tax Records: Document applicable tax law, retention period (typically 8 years)
  • DPIA Finding: Document DPIA result, why processing is deemed necessary, date when decision will be revisited

Exception Notification Letter Template

Subject: Erasure Request Outcome — Partially Granted with Exception — Request ID [ERA-ID]

Dear [Data Principal Name],

We received your erasure request on [DATE] (Request ID: [ERA-ID]). We have reviewed your request and determined that most of your data can be erased, but some data must be retained due to legal requirements.

Data Erased:
• [Data category 1]
• [Data category 2]

Data Retained (Exception):
• [Data category]: Retained until [DATE] due to [legal basis]
Legal Basis: [Explain: e.g., "RBI retention requirement for loan records", "Pending litigation in [Court]"]

Next Steps:
Once the legal requirement expires ([DATE]), we will automatically erase the retained data. You can request a status update at any time by filing another request.

If you believe we are incorrectly retaining data, you may file a grievance with our DPO: [DPO_EMAIL]

Regards,
[Organization Name]

Audit Trail & Documentation

For DPBI compliance verification, you must maintain detailed audit trail of every correction and erasure request.

Correction & Erasure Tracker (Database Table)

Field Description
request_id Unique ID (e.g., COR-2026-04-15-001 or ERA-2026-04-15-001)
principal_id Data Principal who made request
request_type "CORRECTION" or "ERASURE"
data_field For CORRECTION: field name. For ERASURE: data categories
received_date ISO timestamp when request was received
processed_date ISO timestamp when action was taken in own systems
processors_notified_date ISO timestamp when processors were notified
processors_confirmed_date ISO timestamp when all processors confirmed completion
completion_date ISO timestamp when principal was notified of completion
status "PENDING" / "IN_PROGRESS" / "COMPLETED" / "DENIED_EXCEPTION"
exception_reason If DENIED: "LEGAL_HOLD", "RBI_RETENTION", "TAX_RECORD", "DPIA_FINDING" + explanation
processors_list JSON list of processors notified: [{"name": "...", "notified_date": "...", "confirmed_date": "..."}]
authorized_by DPO or compliance officer who authorized action

Implementation Checklist

  • ☐ Process: Correction Form (T-13) and Erasure Form created and published on portal
  • ☐ Identity Verification: Same procedure as DSAR (account login + OTP, VID, email callback, etc.)
  • ☐ Tracker Database: Correction & Erasure Tracker table created with fields listed above
  • ☐ SOP Documented: These 8 steps for erasure, 7 steps for correction documented in training
  • ☐ Processor Register: Data Processing Agreement (T-24) lists all current processors and their contact emails
  • ☐ Notification Templates: Standardized processor notification emails approved by DPO and Legal
  • ☐ Exception Framework: DPO has documented which data categories have legal exceptions and why
  • ☐ Legal Hold Process: Procedure for litigation holds documented; DPO informed of all active litigation
  • ☐ 30-Day Monitoring: Dashboard to track requests approaching 30-day deadline; alerts for escalation
  • ☐ Propagation SLA: Processors must confirm within 5-7 days; escalation if non-responsive
  • ☐ Audit Trail: All corrections/erasures logged with timestamps, authorized by, processors notified
  • ☐ Reporting: Monthly/quarterly report to DPO: requests received, processed, denied with reasons
  • ☐ Training: Grievance team trained on this SOP, correction vs erasure distinction, propagation requirement
