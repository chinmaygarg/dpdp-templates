Access Control Policy
Role-Based Access Control (RBAC) Framework
Version: 1.0 | Effective Date: [DATE] | Status: [Active/Draft/Archived]

1. Principle: Least Privilege & Zero Trust (DPDP Section 8)

DPDP Section 8(3): Organizations must implement role-based access control (RBAC) — no "access all" logins.

Zero Trust Model: Never assume trust after first login. Every request requires: 1. Authentication (who are you? Verify MFA) 2. Authorization (what's your role? Check permissions) 3. Audit (what did you access? Log all access)

2. Role Hierarchy (7 Role Types)

Role Type Data Access Level MFA Required? Audit Trail?
Viewer Read-only dashboards; NO personal data access Standard password ✓ Logged
User (Operational) Customer Service: view assigned customer records; HR: view assigned employee records Standard password ✓ Logged
Power User (Compliance/DPO) Access all personal data for compliance purposes; execute DSAR/deletion ✓ MANDATORY MFA ✓ Detailed log with timestamp
Admin (IT/Database) Database admin: schema changes, user mgmt; elevated system access ✓ MANDATORY MFA ✓ Detailed log; prevent DELETE on audit_log table
DPO Full access for compliance review; cannot delete audit logs ✓ MANDATORY MFA ✓ All actions logged + signed
Security/CISO Monitor audit logs, configure access controls, security audit ✓ MANDATORY MFA ✓ All actions logged
Service Account (Non-Human) Automated jobs: daily delete task, consent sync, SIEM push API key (no password); rotated quarterly ✓ Logged as "system"

3. MFA (Multi-Factor Authentication) Enforcement

MANDATORY MFA for: DPO, Admin, Power User, CISO, any user accessing personal data

Methods (At least 1 of 2 required):
• Time-based OTP (TOTP): Google Authenticator, Microsoft Authenticator
• SMS/Email OTP: 6-digit code sent to registered phone/email

Policy: If MFA not set up within 30 days of hire, access suspended automatically
Recovery: Lost MFA device → DPO manual verification required before re-enabling

4. Quarterly Access Review Process

Every 90 days (Q1, Q2, Q3, Q4):
1. Active User List: Extract all active user accounts + roles from identity management system
2. Verification: For each user, manager confirms: "Is this person still employed in this role?"
3. Stale Access Removal: If no response within 5 days → disable access immediately
4. Role Misalignment: If role changed but system not updated → correct role or remove access
5. Service Account Audit: Verify automated service accounts still needed; no dormant API keys
6. Report Certification: DPO signs off quarterly access review; archive report 7 years

5. Elevated Access (Admin/DPO) — JIT Model (Just-In-Time)

Principle: Admin access is temporary, not permanent. Privilege escalation auto-revokes after 2 hours.

Workflow:
1. User requests admin access in system (must provide justification)
2. Admin approves (or denies) the request
3. System grants elevated access for 2 hours
4. Access auto-revokes at T+2h (even if logged in)
5. Access request logged in audit_log with: who, what, when, justification

Benefit: If admin account compromised, attacker has only 2-hour window before privileges lost.

7. Implementation Checklist

  • ☐ Define 7 role types with access levels (Viewer, User, Power User, Admin, DPO, Security, Service Account)
  • ☐ Implement RBAC in all systems (database, cloud, SaaS applications)
  • ☐ Enforce MFA for DPO, Admin, Power User, CISO roles
  • ☐ Configure JIT elevated access (2-hour auto-revoke for admin privileges)
  • ☐ Create quarterly access review process (90-day cycle, manager verification)
  • ☐ Document all role assignments in access control register
  • ☐ Audit trail: log all access requests, approvals, escalations, and revocations
  • ☐ Train users on MFA setup and JIT escalation process
  • ☐ Test role enforcement: verify Viewer cannot execute deletion, Admin cannot modify audit logs, etc.
  • ☐ Quarterly DPO sign-off on access review; document stale access removed
  • ☐ Annual security audit: penetration test for privilege escalation vulnerabilities
