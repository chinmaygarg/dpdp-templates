1. Breach Definition & Legal Basis (DPDP Section 20)

Data Breach Definition (Section 20): Unauthorized access, use, disclosure, or loss of personal data that compromises confidentiality, integrity, or availability.

Notification SLA (Dual-Timer):
• DPBI (Data Protection Board of India): 72 hours of breach discovery
• CERT-In (Cybersecurity incident): 6 hours if cybersecurity-related (mandatory for ALL incidents)
• Data Principals (Affected persons): Without undue delay if likely to cause harm (Section 20(3))
Missing either SLA = ₹50-100M penalty per incident

2. Incident Classification (4-Type Matrix)

Incident Type Examples DPBI (72hr)? CERT-In (6hr)?
Data Exfiltration Unauthorized download, email export, API dump ✓ YES ✓ YES (if via hacking)
Ransomware/Encryption Data encrypted by malware, extortion demand ✓ YES ✓ YES
Unauthorized Access Brute force login, compromised credentials, lateral movement ✓ YES ✓ YES
Configuration/Human Error Misconfigured S3 bucket, backup exposed, email BCC error ✓ YES (if personal data exposed) ✗ NO (unless hacking involved)

3. Breach Register Schema (14 Fields)

Field Type Purpose
incident_id UUID Unique incident identifier (INC-YYYYMMDD-XXXXX)
discovery_date TIMESTAMP When breach was discovered (triggers 72-hr DPBI SLA)
incident_type ENUM Data Exfiltration | Ransomware | Unauthorized Access | Config Error
severity ENUM CRITICAL | HIGH | MEDIUM | LOW
data_categories_affected TEXT (JSON) ["Aadhaar_VID", "Email", "Phone"]
principals_affected_count INTEGER Number of data principals impacted
likely_harm BOOLEAN Does breach likely cause harm? (triggers data principal notification per Section 20(3))
dpbi_notified_date TIMESTAMP When DPBI was notified (SLA: T+72hr)
cert_in_notified_date TIMESTAMP When CERT-In was notified (SLA: T+6hr for cybersecurity incidents)
principals_notified_date TIMESTAMP When data principals notified (Section 20(3): without undue delay)
root_cause TEXT Investigation finding (compromised credentials, SQL injection, etc.)
remediation_actions TEXT Steps taken to stop incident (isolate systems, revoke keys, patch vulnerability)
post_breach_review_date TIMESTAMP Post-incident review completed (T-23 Post-Breach Review Report)
status ENUM Open | Contained | Remediated | Closed

4. Dual-Timer Notification Timeline

T+0 (Discovery): Incident detected → Incident Commander activated
T+1 (Assessment): Preliminary assessment: data breach? cybersecurity incident?
T+2 (Isolation): Isolate affected systems; begin forensics
T+6 (CERT-In): IF cybersecurity incident → MUST notify CERT-In by now
T+24 (Preliminary DPBI): If breach likely → DPBI preliminary notification with initial assessment
T+72 (Full DPBI): DPBI final notification with full incident details (if not already notified at T+24)
T+0-72 (Data Principal): If likely harm → notify affected principals without undue delay (parallel path)
T+30 (Post-Breach Review): Complete post-incident review; document learnings; implement preventive controls

5. SLA Monitoring & Escalation

Metric Target Yellow Alert Red Alert
CERT-In (6-hour, cybersecurity only) 100% ≤6h Pending at 4h Missed SLA
DPBI (72-hour) 100% ≤72h Pending at 48h Missed SLA = ₹50-100M penalty
Data Principal Notification 100% within 24h (if harm likely) Pending at 12h Missed SLA = reputation damage
