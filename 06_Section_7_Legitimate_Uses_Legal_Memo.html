Section 7 Legitimate Uses Legal Memo
DPDP Act 2023 — Processing Without Consent (FinTech/BFSI Edition)
DPDP Section: Section 7 (Certain Legitimate Uses) | Version: 1.0 | Effective Date: [DATE]
Applicability: Organizations using employment grounds, financial assessment, or legal compliance justifications

Overview

Section 7 of the DPDP Act 2023 permits processing of personal data without explicit consent in nine specific categories. This memo documents which Section 7 legitimate uses apply to your organization and the legal justification for each use case.

⚠️ Important: This memo must be prepared by Legal/Compliance and reviewed by your organization's DPO (if SDF) or external counsel. Using Section 7 without documented justification is non-compliant.

FinTech/BFSI Context: Many fintech and NBFC operations rely on Section 7 grounds, including employee background verification, credit assessment, KYC/AML compliance, and fraud detection. This memo helps document those legitimate uses.
---

Section 7 Legitimate Uses — Nine Categories

The DPDP Act permits processing without consent ONLY if it falls into ONE of these nine categories. Check which apply to your organization:

Category Description FinTech Examples
(i) Employment Processing for employment-related purposes or safeguarding employer interests ✅ BGV, payroll, performance
(ii) Legal Obligations Processing to comply with statutory requirements (RBI, ED, PMLA, etc.) ✅ KYC, AML, RBI reporting
(iii) Vital Interests Processing necessary for medical emergency or public health ⚠️ Rare in fintech
(iv) Financial Exemption Assessment of borrower's creditworthiness if in default ✅ Loan default review
(v) Voluntary Provision Data voluntarily provided for purpose; no objection raised ✅ Application forms
(vi) Government Services Processing by government agencies for benefits/services ❌ Not applicable to private fintech
(vii) Public Function Processing by entity authorized by law to perform public function ⚠️ Rare in fintech
(viii) Lawful Contracts Processing necessary for contract performance ✅ Loan servicing, account management
(ix) Safeguard Interests Safeguarding legitimate interests (IP protection, fraud detection, etc.) ✅ Fraud detection, security

💰 USE CASE 3: FINANCIAL SECTOR EXEMPTION — LOAN DEFAULT ASSESSMENT (Section 7(iv))

Legal Ground: Section 7(iv) — "Assessment of financial situation, assets, liabilities of borrower in default"
Applicable To: Banks, NBFCs, FinTech lenders, credit platforms
FinTech Relevance: CRITICAL for lending fintech — allows processing without consent

MEMO: Loan Default Assessment Processing Justification

1. Define the Default Scenario

  • Product Type: [Personal Loan / Mortgage / Credit Card / Business Loan]
  • Default Trigger: [30 days past due / 60 days / 90 days per RBI classification]
  • Assessment Purpose: [Recovery strategy / Risk classification / Restructuring eligibility]
  • Data Needed: [Bank statements, asset ownership, credit history, income documents, employment records]

2. Legal Justification Under Section 7(iv)

Section 7(iv) explicitly permits processing of a defaulted borrower's financial data to assess:

  • Financial situation (income, assets, liabilities)
  • Ability to repay (revised timeline or structured repayment)
  • Collateral/security position

This is a BFSI-specific exemption and does NOT require consent.

3. Data Minimization Limits

  • ☐ Only assess borrower's OWN financial situation (not family/relatives)
  • ☐ Only collect data necessary for recovery/restructuring (not full life history)
  • ☐ Data retention ends when default is cured or loan written off
  • ☐ No use for marketing or other purposes (strict purpose limitation)

4. FinTech Example — Personal Loan Default

Scenario: Customer X has EMI overdue by 45 days on ₹5 lakh personal loan

Permitted Processing (Section 7(iv) — NO CONSENT NEEDED):

  • ✅ Pull credit bureau report (CIBIL/Experian) to assess risk
  • ✅ Request bank statements to assess liquidity
  • ✅ Obtain employer verification to confirm employment
  • ✅ Assess asset ownership (property, vehicles) for collateral
  • ✅ Communicate restructuring options

NOT Permitted:

  • ❌ Use data for cross-sell/marketing personal loans
  • ❌ Share data with third-party collection agency without prior agreement
  • ❌ Retain data beyond loan settlement + 2-year hold

Conclusion: Default assessment is justified under Section 7(iv) as BFSI-specific exemption. No consent required, but purpose limitation strictly enforced.

MEMO: Fraud Detection & Security Processing Justification

1. Identify the Legitimate Interest

  • Interest: [Preventing fraud / Detecting unauthorized transactions / IP protection / Security breach response]
  • Risk: [Financial loss / Data breach / Account takeover / Reputational damage]
  • Data Used: [Transaction patterns / Device fingerprints / Biometric mismatch / Velocity checks]

2. Balancing Test (Must Pass)

Section 7(ix) requires a balancing of interests — organization's interest must NOT be overridden by individual's interests/rights:

  • ☐ Organization's Interest: [Prevent ₹X fraud / Protect Y customers]
  • ☐ Individual's Interest/Right: [Privacy in transaction data / Not being flagged]
  • ☐ Balancing Result: [Organization interest prevails because fraud risk is material / Individual can request review]

3. Proportionality & Safeguards

  • ☐ Fraud monitoring limited to detection/prevention (not profiling for other uses)
  • ☐ Data encrypted or pseudonymized where possible
  • ☐ Human review required before blocking transaction (not auto-decision)
  • ☐ Customer can appeal if transaction incorrectly flagged
  • ☐ No secondary use (data not sold to third parties)

4. FinTech Example — Real-Time Fraud Detection

Use Case: Payment Platform Fraud Monitoring

Legitimate Interest: Prevent account fraud and unauthorized transactions (₹X annual fraud loss)

Data Processed: Real-time transaction monitoring (amount, merchant, location, device, velocity)

Processing Logic: ML model flags high-risk patterns (e.g., ₹50,000 transaction from new device in different country)

Action Taken: Transaction declined pending OTP verification (not silent block)

Customer Rights: Can view flagged transactions, submit appeal, request explanation

Data Retention: Transaction logs kept for 2 years for compliance; deleted after

Conclusion: Fraud monitoring is justified under Section 7(ix) because organization's interest (fraud prevention) outweighs customer's interest in unrestricted transactions, AND proportionality safeguards are in place.

🏦 ACCOUNT AGGREGATOR (AA) FRAMEWORK COMPLIANCE

The RBI Account Aggregator (AA) Framework enables secure, consent-driven financial data sharing between FIUs (Financial Information Users like fintech lenders) and AAs (Account Aggregators). This section clarifies the Fiduciary/FIU obligations, consent model, and data handling requirements under the AA Framework.

What is the AA Framework?

Parties: Data Principal (customer) → Account Aggregator (AA) → Fiduciary/FIU (your fintech)
Data Flow: AA is Processor (licensed by RBI); Fiduciary is Fiduciary/Controller (collects & uses data)
Consent Model: Data Principal grants consent via AA portal (NOT to Fiduciary directly). AA then shares consent artefact with Fiduciary in XML format. Fiduciary receives AA Consent Artefact as proof of valid consent.
Authorization: RBI AA Ecosystem Framework v1.2 (2021); mandatory for all FIUs collecting financial data from AA

Fiduciary Obligations Under AA Framework

Obligation Implementation Requirement DPDP Alignment
1. Request AA Consent Artefact • Customer selects loan product (e.g., ₹5L personal loan)
• Fiduciary app shows: "Link your bank accounts via RBI Account Aggregator to auto-verify income"
• Customer clicks "Link via AA" → redirected to AA portal login
• Customer logs into AA, selects which bank accounts to share (e.g., HDFC & ICICI Savings)
• Customer grants consent in AA portal
• AA returns XML Consent Artefact to Fiduciary (via REST callback)
• Fiduciary stores artefact in encrypted DB field `aa_consent_artefact` (TEXT, AES-256)		 Section 6 consent — but granted via AA, not Fiduciary form. Fiduciary receives proof (Consent Artefact) that consent was FSIUU-valid at AA layer.
2. Link Data via AA Reference ID Only • AA Consent Artefact includes `ReferenceId` (e.g., AA-ORG-2024-1234567890)
• CRITICAL: Fiduciary must link account data using ONLY the AA Reference ID
• NEVER store or reference raw bank account numbers (e.g., 1234567890123456)
• Db schema: Store (account_reference_id TEXT, aa_reference_id TEXT, bank_name TEXT)
• Masking: In API responses, return only "Account ****7890 from HDFC" (masked)
• Audit trail: Log all AA-sourced data with aa_reference_id, not account number		 Data minimisation (Section 5 DPDP). No secondary use of raw bank account numbers. AA Reference ID is privacy-safe linkage mechanism.
3. Store AA Consent Artefact (Immutable) • Artefact is XML/JSON document signed by AA (digital signature inside)
• Store in DB field: `aa_consent_artefact` BLOB/TEXT
• NO modifications; treat as immutable audit record
• Retention: 7 years minimum (AA Data Security standard; aligns with DPDP breach archive period)
• Encryption: At-rest AES-256; in-transit TLS 1.3
• Backup: Archive to WORM storage after 90 days (same as audit logs T-35)
• Verification: Hash-chain the artefact in audit_log table (include sha256_hash of artefact)		 Section 5(b) DPDP — Fiduciary is accountable for data receipt. Consent Artefact is proof of consent validity. Breach of Artefact storage = Section 8(1) violation.
4. Data Minimisation — Limit AA Usage to Stated Purpose • AA Consent Artefact specifies FIU Purpose: e.g., "Credit Decision for Personal Loan ₹5L"
• Fiduciary MUST NOT use AA-linked data for:
❌ Cross-sell (marketing other products)
❌ Analytics (secondary use)
❌ Sharing with third parties (collectors, credit bureau)
❌ Profiling (credit scoring beyond stated loan purpose)
• Enforcement: Implement access control rules in code:
• Flag any query that uses aa_reference_id data outside stated purpose
• Audit log triggers warning if AA-data used for non-loan decision (auto-escalate to DPO)		 Section 5 (Data minimisation), Section 7(viii) (contract performance purpose limits). AA data is for loan decisioning only; cannot be repurposed.
5. Secure Data Transmission from AA • AA delivers account data via secure REST API (Fiduciary acts as FIU endpoint)
• TLS 1.3 minimum; certificate pinning recommended
• Payload encryption: Data encrypted by AA using FIU public key (RSA-2048)
• Fiduciary decrypts using private key (kept in HSM or secure vault)
• All API calls logged (timestamp, AA IP, payload size, status code) in audit_log
• Rate limiting: Max 10 AA data fetch requests per minute per org (fraud detection)		 Section 8 (Information Security). AA data is financial — highest encryption standard required.
6. Implement AA Withdrawal / Data Deletion Path • AA Workflow: Data Principal revokes consent in AA portal → AA notifies Fiduciary (callback)
• Fiduciary receives revocation notice with aa_reference_id
• Obligation: Delete/anonymize all data linked to that aa_reference_id within 30 days
• Implementation:
• Mark aa_consent_artefact as "revoked" with timestamp
• Delete/anonymize all bank account balances, transaction history linked to that AA ref
• Propagate deletion request to all sub-processors (credit bureau, analytics vendor)
• Log deletion in audit_log with reason "AA consent revoked"
• Send confirmation email to data principal within 24 hours		 Section 12(4) DPDP — Fiduciary must propagate deletion to processors. AA revocation = mandatory erasure request.
7. Audit & Compliance Reporting • Monthly Report: Count active AA consents, revocations, data uses per purpose
• Quarterly Report to DPO: List all AA data sources, deletion completions, incidents
• Annual Audit: Verify all AA Consent Artefacts are intact; test deletion workflows
• DPBI Audit: Provide AA data register on demand (showing aa_reference_id usage patterns)
• Data Subject Request (DSAR): If principal requests "show my AA-sourced data", pull from aa_reference_id bucket; return in CSV with account numbers masked		 Section 5(c) DPDP — Fiduciary must demonstrate compliance with AA Framework via audit trail.

AA Data Handling in Fintech Loan Decisioning

FinTech Example: Income Verification via AA

Scenario: Customer applies for ₹5L personal loan. Fintech wants to auto-verify income from bank statements.

Old Way (Pre-AA) — Manual Documents:
Customer uploads salary slips → Fintech manual review → Takes 3 days
Problem: Salary slips can be forged; no DPDP consent clarity

New Way (AA Framework):

  1. Customer Flow:
    • Fintech app shows: "Link your bank account to auto-verify income in 30 seconds"
    • Customer clicks "Link with Account Aggregator"
    • Browser redirects to AA login page
    • Customer logs into AA with credentials
    • AA shows banks already linked (e.g., HDFC, ICICI)
    • Customer selects HDFC Savings account → confirms consent in AA
    • AA portal shows: "Account Aggregator will share your bank statement data with [Fintech] for loan decision only. This will NOT be used for marketing. Your data stays encrypted."
    • Customer clicks "Confirm"
  2. Backend Flow (Fintech):
    • AA sends REST callback with XML Consent Artefact containing:
    - `ConsentId`: UNIQ consent ID
    - `FIUId`: Fintech's AA-assigned ID
    - `ReferenceId`: AA-ORG-2024-1234567890 (bank account reference)
    - `DateFrom`: "2023-01-01", `DateTo`: "2024-01-31" (12 months data)
    - `Digital Signature`: (AA's signing key — verify before accepting)
    • Fintech stores artefact in DB (encrypted)
    • Fintech calls AA API: "Get data for ReferenceId AA-ORG-2024-1234567890"
    • AA returns encrypted payload: {"transactions": [...], "balance": 250000, "accountNumber": "[encrypted]"}
    • Fintech decrypts, analyzes transaction history → detects 6-month average monthly credit: ₹45,000
    • Score: ✅ Income verified → Auto-approve ₹5L loan application
  3. Data Handling Post-Verification:
    • Store: aa_reference_id: "AA-ORG-2024-1234567890", verified_income: 45000, decision_date: "2024-01-15"
    • NEVER store: Raw bank account number, transaction list, or customer name (from bank payload)
    • Delete: After 90 days or loan settlement + 2 years (whichever earlier)
    • Audit log entry: `{ event: 'AA_DATA_USED', aa_reference_id: '...', purpose: 'LOAN_CREDIT_DECISION', decision: 'APPROVE', timestamp: '...' }`
  4. Withdrawal Scenario:
    • Customer changes mind 10 days later, revokes consent in AA portal
    • AA notifies Fintech: `{ consentId: '...', status: 'REVOKED' }`
    • Fintech marks aa_consent_artefact as revoked; deletes verified_income data; sends SMS to customer: "AA consent revoked. Loan data deleted within 30 days."
    • Compliance check: Audit log confirms deletion by Day 30

DPDP Compliance Outcome:
✅ Consent is explicit, informed (AA disclosure shown to customer), unconditional (can revoke anytime)
✅ Purpose is specific (loan decision only — AA artefact states it)
✅ Data minimisation (only 12-month bank data, not full history)
✅ Audit trail (every AA-sourced decision logged with aa_reference_id)
✅ Withdrawal & deletion implemented (30-day SLA)

AA Compliance Checkpoint: If your fintech collects financial data via AA, you must document:
1. AA Consent Artefact storage & encryption
2. AA Reference ID linking (no raw account numbers)
3. Purpose limitation (loan decision only; no marketing)
4. Deletion workflow (30-day revocation path)
5. Sub-processor notification (if sharing AA-sourced data with credit bureau)
Failure to implement = DPDP Section 5(b) accountability violation + RBI AA Framework breach (separate penalty).
